02 검증 방식

이슈 대응 루틴

  1. 동일 조건에서 재현을 먼저 시도해 일시적 문제인지 확인
  2. 발생 조건을 바꿔가며 문제 범위를 좁혀나감
  3. 로그 · 캡처 · 영상으로 기록, 디바이스 · OS · 제품 버전 정리
  4. 재현 절차와 함께 이슈 등록 및 공유

테스트 우선순위 기준

  • 사용자 영향도 — 자주 쓰이거나 오류 시 이용이 막히는 기능 우선
  • 서비스 흐름 — 중간에 끊기면 안 되는 핵심 시나리오 중심
  • 변경 범위 — 이번 릴리즈 수정·추가 영역과 사이드 이펙트 검증

04 프로젝트
PROJECT 01 ／ 개인 프로젝트AUTOMATION

모바일 E2E 자동화 프레임워크 · Appium + WebdriverIO

수동 회귀 검증을 자동화로 옮기기 위해, 실제 앱(SauceLabs My Demo App)을 대상으로 레이어를 분리한 POM 프레임워크를 직접 설계한 프로젝트. 로그인부터 결제까지 6개 화면을 E2E로 검증합니다.

  • POM(Page Object Model) 기반 6개 화면 35개 테스트케이스 (로그인 · 상품 · 상세 · 장바구니 · 결제 · 로그아웃, TC-ID 부여)
  • 레이어 분리 설계 — spec(무엇을 검증) / Page(화면 조작) / selectors(locator 중앙화) / helpers(생명주기 · 공통 흐름) / data(테스트 데이터)
  • 테스트 독립성 · 명시적 대기 — 전역 상태 없이 각 테스트가 동일한 시작 상태 보장, BasePage에 대기 내장
  • 환경 전환 — .env 값만 바꿔 디바이스 · 앱 교체, 클라우드 디바이스팜(BrowserStack 등)까지 확장 가능한 구조
  • Allure 리포트 · GitHub Actions CI 구성, Appium MCP로 앱 탐색 · 버그리포트 자동화 실험
